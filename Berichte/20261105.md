# Übung: Analyse und Symbolic Mode

## Ziel der Übung

In dieser Übung wurden grundlegende Linux-Dateiberechtigungen analysiert und
mit dem symbolischen Modus von `chmod` gezielt verändert. Dabei wurden
Berechtigungen für User, Group und Others einzeln sowie kombiniert angepasst.

---

## Aufgabe 1: Rechte auslesen

Die Permissions, der User Owner und der Group Owner der Datei
`/tmp/lab/wrestle_plans.txt` wurden mit folgendem Befehl ermittelt:

```sh
ls -l /tmp/lab/wrestle_plans.txt
```

```sh
-rwxr-xr-x 1 root root 6 May 11 06:08 /tmp/lab/wrestle_plans.txt
```

Die Datei gehört dem User `root` und der Gruppe `root`. Der User besitzt alle
Rechte (rwx), Group und Others dürfen die Datei lesen und ausführen (r-x),
jedoch nicht schreiben.

---

## Aufgabe 2: Private Files

Es wurde eine neue Datei `notes.txt` erstellt und der Others-Klasse das
Read-Recht entzogen:

```sh
touch notes.txt
chmod o-r notes.txt
ls -l notes.txt
```

```sh
-rw-rw---- 1 kali kali 0 May 11 06:10 notes.txt
```

Durch `chmod o-r` wurde das Leserecht für Others entfernt. User und Group
behalten ihre bisherigen Rechte (rw).

---

## Aufgabe 3: Group Sharing

Die Permissions von `notes.txt` wurden so gesetzt, dass der User alle Rechte
(rwx) behält, die Group lesen und schreiben (rw) darf und Others keinerlei
Zugriff haben. Dies erfolgte mit einem einzigen `chmod`-Aufruf:

```sh
chmod u=rwx,g=rw,o= notes.txt
ls -l notes.txt
```

```sh
-rwxrw---- 1 kali kali 0 May 11 06:10 notes.txt
```

Die Permissions wurden exakt wie gefordert gesetzt. Der erste Versuch mit
`u=rwx, g=rw, o=` (mit Leerzeichen) schlug fehl, da `chmod` keine Leerzeichen
innerhalb des Modus akzeptiert.

---

## Aufgabe 4: Skript-Vorbereitung

Eine neue Datei `myscript.sh` wurde erstellt und dem User sowie der Group das
Ausführungsrecht vergeben:

```sh
touch myscript.sh
chmod ug+x myscript.sh
ls -l myscript.sh
```

```sh
-rwxrwxr-- 1 kali kali 0 May 11 06:13 myscript.sh
```

Mit `ug+x` wurde das Execute-Recht für User und Group hinzugefügt, ohne die
bestehenden Rechte zu verändern. Ein erster Versuch scheiterte aufgrund eines
Tippfehlers (`myscripts.sh` statt `myscript.sh`).

---

## Aufgabe 5: Kollektive Änderung

Eine neue Datei `data.txt` wurde erstellt. Anschließend wurden der Group und
Others mit einem einzigen Befehl die Write- und Execute-Rechte entzogen:

```sh
touch data.txt
chmod go-wx data.txt
ls -l data.txt
```

```sh
-rw-r--r-- 1 kali kali 0 May 11 06:14 data.txt
```

Durch `go-wx` wurden Write und Execute für Group und Others gleichzeitig
entfernt. Das Read-Recht bleibt für beide Klassen erhalten.

---

## Zusammenfassung

Der symbolische Modus von `chmod` ermöglicht eine präzise und lesbare
Steuerung von Dateiberechtigungen. Durch die Operatoren `+`, `-` und `=`
lassen sich Rechte gezielt hinzufügen, entfernen oder exakt setzen — sowohl
für einzelne Klassen als auch kombiniert für mehrere gleichzeitig.

# Übung: Ownership und Gruppenkollaboration

## Ziel der Übung

In dieser Übung wurden Dateibesitz und Gruppenzugehörigkeit unter Linux
untersucht und gezielt verändert. Dabei wurden die Befehle `chown`, `chgrp`
und `su` verwendet sowie der Unterschied zwischen `su` und `su -` praktisch
demonstriert.

---

## Aufgabe 1: Root Shell & sudoers

Mit `sudo su` wurde in eine Root-Shell gewechselt. Anschließend wurde der
User `blondie` über `visudo` in die sudoers-Datei eingetragen, sodass er den
Befehl `mv` als root ausführen darf:

```sh
sudo su
visudo
```

Eingetragene Zeile:
blondie ALL=(root) /bin/mv

---

## Aufgabe 2: Group Change

Als User `blondie` wurde die Datei `song.txt` im Home-Verzeichnis erstellt
und anschließend mit `sudo mv` nach `/tmp/lab/` verschoben. Danach wurde der
Group Owner auf `music` geändert:

```sh
su - blondie
touch ~/song.txt
sudo mv ~/song.txt /tmp/lab/
chgrp music /tmp/lab/song.txt
ls -l /tmp/lab/song.txt
```

```sh
-rw-rw-r-- 1 blondie music 0 May 11 06:21 /tmp/lab/song.txt
```

Der Group Owner wurde erfolgreich auf `music` gesetzt. Das Verschieben über
`sudo mv` war notwendig, da `blondie` keine direkten Schreibrechte in
`/tmp/lab/` besitzt.

---

## Aufgabe 3: Unterschied su und su -

Der Unterschied zwischen `su` und `su -` wurde anhand des aktuellen
Arbeitsverzeichnisses demonstriert:

```sh
su blondie
pwd
exit
```

```sh
/home/kali
```

```sh
su - blondie
pwd
exit
```

```sh
/home/blondie
```

`su blondie` übernimmt lediglich die Identität des Users, bleibt aber im
aktuellen Verzeichnis. `su - blondie` simuliert einen vollständigen Login und
wechselt ins Home-Verzeichnis des Ziel-Users.

---

## Aufgabe 4: User Transfer

Als root wurde der User Owner der Datei `/tmp/lab/song.txt` auf `elvis`
geändert:

```sh
chown elvis /tmp/lab/song.txt
ls -l /tmp/lab/song.txt
```

```sh
-rw-rw-r-- 1 elvis music 0 May 11 06:21 /tmp/lab/song.txt
```

Der User Owner wurde erfolgreich von `blondie` auf `elvis` übertragen. Der
Group Owner `music` blieb unverändert.

---

## Aufgabe 5: Wrestler-Szenario

Zunächst wurde `hogan` analog zu `blondie` in die sudoers-Datei eingetragen:

```sh
visudo
```
hogan ALL=(root) /bin/mv

Anschließend wurde als User `hogan` die Datei `training.txt` erstellt,
nach `/tmp/lab/` verschoben, die Gruppe auf `wrestle` gesetzt und die
Berechtigungen angepasst:

```sh
su - hogan
touch ~/training.txt
sudo mv ~/training.txt /tmp/lab/
chgrp wrestle /tmp/lab/training.txt
chmod g=rw,o= /tmp/lab/training.txt
ls -l /tmp/lab/training.txt
```

```sh
-rw-rw---- 1 hogan wrestle 0 May 11 06:34 /tmp/lab/training.txt
```

Die Group `wrestle` hat Lese- und Schreibrechte, Others haben keinen Zugriff.
Ein direktes `touch /tmp/lab/training.txt` als `hogan` schlug fehl, da das
Verzeichnis `/tmp/lab/` root gehört und keine Schreibrechte für andere User
erlaubt.

---

## Aufgabe 6: Einschränkungen

Als User `elvis` wurde versucht, den Group Owner der Datei
`/tmp/lab/song.txt` auf `emperors` zu ändern:

```sh
su - elvis
chgrp emperors /tmp/lab/song.txt
```

```sh
chgrp: changing group of '/tmp/lab/song.txt': Operation not permitted
```

Der Befehl schlug fehl, da `elvis` zwar User Owner der Datei ist, jedoch
nicht Mitglied der Gruppe `emperors`. Unter Linux darf ein User den Group
Owner einer Datei nur auf eine Gruppe ändern, der er selbst angehört.

---

## Zusammenfassung

Ownership und Gruppenzugehörigkeit sind zentrale Konzepte der
Linux-Zugriffskontrolle. `chown` und `chgrp` erlauben gezielte Übertragung
von Dateibesitz, jedoch nur innerhalb der eigenen Gruppenmitgliedschaften.
Der Unterschied zwischen `su` und `su -` zeigt, dass eine vollständige
Login-Session die Umgebungsvariablen und das Arbeitsverzeichnis des Ziel-Users
übernimmt, während ein einfacher `su`-Aufruf nur die Identität wechselt.

# Übung: Directory Permissions und Octal Notation

## Ziel der Übung

In dieser Übung wurden Dateiberechtigungen in oktaler Notation gesetzt sowie
die besonderen Eigenschaften von Verzeichnis-Permissions untersucht.
Insbesondere wurde der Unterschied zwischen dem Execute-Bit bei Dateien und
bei Verzeichnissen erarbeitet.

---

## Aufgabe 1: Oktale Umwandlung

Die oktalе Notation fasst die drei Permission-Bits (r=4, w=2, x=1) jeder
Klasse zu einer Ziffer zusammen. Die drei Zielzustände wurden wie folgt
umgewandelt:

```sh
chmod 755 /tmp/lab/wrestle_plans.txt
chmod 660 notes.txt
chmod 700 myscript.sh
```

| Permission  | User | Group | Others | Oktal |
|-------------|------|-------|--------|-------|
| rwxr-xr-x   | 7    | 5     | 5      | 755   |
| rw-rw----   | 6    | 6     | 0      | 660   |
| rwx------   | 7    | 0     | 0      | 700   |

---

## Aufgabe 2: Pub-Verzeichnis

Das Verzeichnis `/home/kali/pub` wurde erstellt und mit einem einzigen
Befehl in oktaler Notation berechtigt. Der Besitzer darf alles (7), die
Gruppe darf auflisten und wechseln (5), Others dürfen nur wechseln (1):

```sh
mkdir /home/kali/pub
chmod 751 /home/kali/pub
ls -ld /home/kali/pub
```

```sh
drwxr-x--x 2 root root 4096 May 11 06:45 /home/kali/pub
```

Die Permissions wurden korrekt gesetzt. Others können mit `cd` in das
Verzeichnis wechseln, jedoch dessen Inhalt nicht mit `ls` auflisten.

---

## Aufgabe 3: Recursive Lockdown

Das Verzeichnis `/tmp/lab/drafts/` und alle darin enthaltenen Dateien wurden
rekursiv so berechtigt, dass die Gruppe `music` lesen und schreiben darf,
Others jedoch keinen Zugriff haben:

```sh
chmod -R g=rw,o= /tmp/lab/drafts/
ls -l /tmp/lab/drafts/
```

```sh
total 4
-rw-rw---- 1 root root 6 May 11 06:08 lyrics.txt
```

Durch den `-R` Flag wurden die Permissions rekursiv auf das Verzeichnis und
alle enthaltenen Dateien angewendet.

---

## Aufgabe 4: Search Permission

Damit andere User zwar nicht den Inhalt von `/home/kali` auflisten, aber
in das bereits existierende Unterverzeichnis `/home/kali/pub` wechseln
können, wurde das Execute-Bit für Others gesetzt:

```sh
chmod o=x /home/kali
ls -ld /home/kali
```

```sh
drwx——x 22 kali kali 4096 May 11 06:45 /home/kali
```

Das Read-Bit für Others wurde entfernt, sodass `ls` im Verzeichnis nicht
möglich ist. Das Execute-Bit erlaubt jedoch weiterhin den Zugriff auf
bekannte Unterverzeichnisse wie `/home/kali/pub` per `cd`.

---

## Aufgabe 5: Analyse

Der Befehl `chmod -R 660 /tmp/lab/` wurde ausgeführt und die Auswirkungen
untersucht:

```sh
chmod -R 660 /tmp/lab/
ls -l /tmp/lab/
```

```sh
total 4
drw-rw---- 2 root root   60 May 11 06:08 drafts
drw-rw---- 2 root root   40 May 11 06:08 gigs
-rw-rw---- 1 elvis music  0 May 11 06:21 song.txt
-rw-rw---- 1 hogan wrestle 0 May 11 06:34 training.txt
-rw-rw---- 1 root root    6 May 11 06:08 wrestle_plans.txt
```

Die Unterverzeichnisse `drafts` und `gigs` zeigen nun `drw-rw----`. Das
Execute-Bit fehlt vollständig. Bei Verzeichnissen bedeutet das Execute-Bit
nicht "ausführen", sondern "betreten" — ohne es kann kein User mehr mit `cd`
in die Unterverzeichnisse wechseln. `chmod -R 660` hat dieses Bit rekursiv
entfernt. Der Test mit `cd /tmp/lab/drafts` war als root möglich, da root
grundsätzlich keinen Zugriffsbeschränkungen unterliegt. Für normale User
würde der Befehl mit "Permission denied" fehlschlagen.

---

## Zusammenfassung

Die oktale Notation bietet eine kompakte Möglichkeit, Berechtigungen präzise
zu setzen. Bei Verzeichnissen hat das Execute-Bit eine besondere Bedeutung:
Es kontrolliert nicht die Ausführung, sondern das Betreten des Verzeichnisses.
Fehlt es, ist `cd` selbst dann nicht möglich, wenn Lese- und Schreibrechte
vorhanden sind. Rekursive Änderungen mit `-R` müssen daher mit Bedacht
eingesetzt werden.

# Übung: Webserver Directory Permissions

## Ziel der Übung

In dieser Übung wurden die Auswirkungen des Read- und Execute-Bits auf
Verzeichnisse am Beispiel des Apache-Webserver-Verzeichnisses `/var/www`
untersucht. Dabei wurde der Unterschied zwischen dem `r`- und dem `x`-Flag
bei Verzeichnissen praktisch demonstriert.

---

## Ausgangszustand

Zunächst wurden die bestehenden Permissions von `/var/www` ausgelesen:

```sh
ls -ld /var/www
```

```sh
drwxr-xr-x 3 root root 4096 May 29 2025 /var/www
```

Im Ausgangszustand hat Others sowohl Lese- (r) als auch Execute-Rechte (x)
auf das Verzeichnis.

---

## Teil 1: other/read entfernen

Das Read-Bit für Others wurde entfernt:

```sh
chmod o-r /var/www
ls -ld /var/www
```

```sh
drwxr-x--x 3 root root 4096 May 29 2025 /var/www
```

Anschließend wurde als User `kali` getestet:

```sh
su kali
ls /var/www
```

```sh
ls: cannot open directory '/var/www': Permission denied
```

```sh
cd /var/www
cd html
pwd
```

```sh
/var/www/html
```

**Auswertung:** Ohne das `r`-Flag kann `kali` den Inhalt von `/var/www`
nicht auflisten — `ls` schlägt fehl. Da das `x`-Flag jedoch noch gesetzt
ist, kann das Verzeichnis betreten werden. Ein Wechsel in das bekannte
Unterverzeichnis `html` ist daher weiterhin möglich.

---

## Teil 2: other/read zurück, other/execute entfernen

Das Read-Bit wurde wiederhergestellt und das Execute-Bit für Others entfernt:

```sh
chmod o+r /var/www
chmod o-x /var/www
ls -ld /var/www
```

```sh
drwxr-xr-- 3 root root 4096 May 29 2025 /var/www
```

Test als User `kali`:

```sh
su kali
ls /var/www
```

```sh
ls: cannot access '/var/www/html': Permission denied
html
```

```sh
cd /var/www
```

```sh
cd: permission denied: /var/www
```

```sh
pwd
```

```sh
/home/kali
```

**Auswertung:** Mit dem `r`-Flag kann `kali` den Inhalt von `/var/www`
zwar auflisten und sieht den Eintrag `html`. Da das `x`-Flag jedoch fehlt,
ist es weder möglich, in `/var/www` zu wechseln, noch auf die
Unterverzeichnisse zuzugreifen. `ls` zeigt zwar den Namen, kann aber keine
Details abrufen.

---

## Zusammenfassung

Das Experiment zeigt den grundlegenden Unterschied zwischen dem `r`- und
dem `x`-Flag bei Verzeichnissen:

| Flag | Bedeutung bei Verzeichnissen |
|------|------------------------------|
| `r`  | Inhalt auflisten (`ls`)      |
| `x`  | Verzeichnis betreten (`cd`)  |

Beide Flags sind unabhängig voneinander. Fehlt `r`, kann der Inhalt nicht
aufgelistet werden, das Betreten und der Zugriff auf bekannte Unterverzeichnisse
bleiben aber möglich. Fehlt `x`, ist weder `cd` noch der Zugriff auf
Unterverzeichnisse möglich, selbst wenn der Name per `ls` sichtbar ist.

---
