# Übung 1: Analyse und Symbolic Mode

Permissions wurden mit dem symbolischen Modus von `chmod` analysiert und gezielt verändert.

## Rechte auslesen

```sh
ls -l /tmp/lab/wrestle_plans.txt
```

```sh
-rwxr-xr-x 1 root root 6 May 11 06:08 /tmp/lab/wrestle_plans.txt
```

Gehört `root:root`. User hat alle Rechte (rwx), Group und Others dürfen lesen und ausführen (r-x).

---

## Private Files

```sh
touch notes.txt
chmod o-r notes.txt
ls -l notes.txt
```

```sh
-rw-rw---- 1 kali kali 0 May 11 06:10 notes.txt
```

Das Leserecht für Others wurde entfernt, User und Group behalten ihre Rechte.

---

## Group Sharing

```sh
chmod u=rwx,g=rw,o= notes.txt
ls -l notes.txt
```

```sh
-rwxrw---- 1 kali kali 0 May 11 06:10 notes.txt
```

Ein erster Versuch mit Leerzeichen (`u=rwx, g=rw, o=`) schlug fehl — `chmod` akzeptiert keine Leerzeichen im Modus.

---

## Skript-Vorbereitung

```sh
touch myscript.sh
chmod ug+x myscript.sh
ls -l myscript.sh
```

```sh
-rwxrwxr-- 1 kali kali 0 May 11 06:13 myscript.sh
```

Ein erster Versuch scheiterte durch einen Tippfehler (`myscripts.sh`).

---

## Kollektive Änderung

```sh
touch data.txt
chmod go-wx data.txt
ls -l data.txt
```

```sh
-rw-r--r-- 1 kali kali 0 May 11 06:14 data.txt
```

Write und Execute wurden für Group und Others gleichzeitig entfernt, das Read-Recht bleibt erhalten.

# Übung 2: Ownership und Gruppenkollaboration

## sudoers – blondie

```sh
sudo su
visudo
```

Eingetragene Zeile:

```
blondie ALL=(root) /bin/mv
```

---

## Group Change

```sh
su - blondie
touch ~/song.txt
sudo mv ~/song.txt /tmp/lab/
chgrp music /tmp/lab/song.txt
ls -l /tmp/lab/song.txt
```

```sh
-rw-rw-r-- 1 blondie music 0 May 11 06:21 /tmp/lab/song.txt
```

`sudo mv` war nötig, da `blondie` keine direkten Schreibrechte in `/tmp/lab/` hat.

---

## Unterschied su und su -

```sh
su blondie
pwd
exit
```

```sh
/home/kali
```

```sh
su - blondie
pwd
exit
```

```sh
/home/blondie
```

`su` wechselt nur die Identität, bleibt im aktuellen Verzeichnis. `su -` simuliert einen vollständigen Login und wechselt ins Home-Verzeichnis des Ziel-Users.

---

## User Transfer

```sh
chown elvis /tmp/lab/song.txt
ls -l /tmp/lab/song.txt
```

```sh
-rw-rw-r-- 1 elvis music 0 May 11 06:21 /tmp/lab/song.txt
```

---

## Wrestler-Szenario

`hogan` wurde analog zu `blondie` in sudoers eingetragen (`hogan ALL=(root) /bin/mv`), dann:

```sh
su - hogan
touch ~/training.txt
sudo mv ~/training.txt /tmp/lab/
chgrp wrestle /tmp/lab/training.txt
chmod g=rw,o= /tmp/lab/training.txt
ls -l /tmp/lab/training.txt
```

```sh
-rw-rw---- 1 hogan wrestle 0 May 11 06:34 /tmp/lab/training.txt
```

Ein direktes `touch /tmp/lab/training.txt` schlug fehl, da `/tmp/lab/` root gehört.

---

## Einschränkungen

```sh
su - elvis
chgrp emperors /tmp/lab/song.txt
```

```sh
chgrp: changing group of '/tmp/lab/song.txt': Operation not permitted
```

`elvis` ist nicht Mitglied der Gruppe `emperors`. Ein User darf den Group Owner nur auf eine Gruppe ändern, der er selbst angehört.

# Übung 3: Directory Permissions und Octal Notation

## Oktale Umwandlung

| Permission  | User | Group | Others | Oktal |
|-------------|------|-------|--------|-------|
| rwxr-xr-x   | 7    | 5     | 5      | 755   |

| rwx------   | 7    | 0     | 0      | 700   |

```sh
chmod 755 /tmp/lab/wrestle_plans.txt
chmod 660 notes.txt
chmod 700 myscript.sh
```

---

## Pub-Verzeichnis

```sh
mkdir /home/kali/pub
chmod 751 /home/kali/pub
ls -ld /home/kali/pub
```

```sh
drwxr-x--x 2 root root 4096 May 11 06:45 /home/kali/pub
```

Others können mit `cd` wechseln, aber den Inhalt nicht mit `ls` auflisten.

---

## Recursive Lockdown

```sh
chmod -R g=rw,o= /tmp/lab/drafts/
ls -l /tmp/lab/drafts/
```

```sh
-rw-rw---- 1 root root 6 May 11 06:08 lyrics.txt
```

---

## Search Permission

```sh
chmod o=x /home/kali
ls -ld /home/kali
```

```sh
drwx——x 22 kali kali 4096 May 11 06:45 /home/kali
```

`ls` im Verzeichnis ist für Others nicht möglich, `cd` in bekannte Unterverzeichnisse jedoch schon.

---

## Analyse: chmod -R 660

```sh
chmod -R 660 /tmp/lab/
ls -l /tmp/lab/
```

```sh
drw-rw---- 2 root root    60 May 11 06:08 drafts
drw-rw---- 2 root root    40 May 11 06:08 gigs
-rw-rw---- 1 elvis music   0 May 11 06:21 song.txt
-rw-rw---- 1 hogan wrestle 0 May 11 06:34 training.txt
-rw-rw---- 1 root root     6 May 11 06:08 wrestle_plans.txt
```

Das Execute-Bit fehlt bei allen Einträgen. Bei Verzeichnissen bedeutet `x` nicht "ausführen" sondern "betreten" — ohne es ist `cd` in Unterverzeichnisse nicht mehr möglich. Als root war der Test trotzdem erfolgreich, da root keinen Zugriffsbeschränkungen unterliegt.

# Übung 4: Webserver Directory Permissions

## Teil 1: other/read entfernen

```sh
chmod o-r /var/www
ls -ld /var/www
```

```sh
drwxr-x--x 3 root root 4096 May 29 2025 /var/www
```

```sh
su kali
ls /var/www
```

```sh
ls: cannot open directory '/var/www': Permission denied
```

```sh
cd /var/www
cd html
pwd
```

```sh
/var/www/html
```

Ohne `r` kann `kali` den Inhalt nicht auflisten. Da `x` noch gesetzt ist, kann das Verzeichnis betreten und in `html` gewechselt werden.

---

## Teil 2: other/read zurück, other/execute entfernen

```sh
chmod o+r /var/www
chmod o-x /var/www
ls -ld /var/www
```

```sh
drwxr-xr-- 3 root root 4096 May 29 2025 /var/www
```

```sh
su kali
ls /var/www
```

```sh
ls: cannot access '/var/www/html': Permission denied
html
```

```sh
cd /var/www
```

```sh
cd: permission denied: /var/www
```

Der Verzeichnisname `html` ist sichtbar, Details können aber nicht abgerufen werden. Ohne `x` ist weder `cd` noch der Zugriff auf Unterverzeichnisse möglich.

| Flag | Bedeutung bei Verzeichnissen |
|------|------------------------------|
| `r`  | Inhalt auflisten (`ls`)      |
| `x`  | Verzeichnis betreten (`cd`)  |

# Übung 5: Webserver Directory Permissions II

## Grundkonfiguration

```sh
mkdir /var/www/data
chown root:www-data /var/www/data
chmod 750 /var/www/data
chmod o+x /var/www/data
ls -ld /var/www/data
```

```sh
drwxr-x--x 2 root www-data 4096 May 11 07:04 /var/www/data
```

`www-data` kann lesen und wechseln, Others nur wechseln.

---

## Sticky Bit

```sh
chmod +t /var/www/data
ls -ld /var/www/data
```

```sh
drwxr-x--t 2 root www-data 4096 May 11 07:04 /var/www/data
```

Das `t` zeigt das Sticky Bit. Jeder User darf nur seine eigenen Dateien löschen.

---

## SGID Bit

```sh
chmod g+s /var/www/data
ls -ld /var/www/data
```

```sh
drwxr-s--t 2 root www-data 4096 May 11 07:04 /var/www/data
```

Das `s` an der Group-Execute-Stelle zeigt das SGID Bit.

---

## Test: Automatische Gruppenzuweisung

```sh
touch /var/www/data/testfile.txt
ls -l /var/www/data
```

```sh
-rw-r--r-- 1 root www-data 0 May 11 07:08 testfile.txt
```

Die Datei wurde automatisch der Gruppe `www-data` zugeordnet — das SGID Bit hat die Gruppe korrekt vererbt.

| Mechanismus     | Wirkung                                          |
|-----------------|--------------------------------------------------|
| `chmod 750`+`o+x` | Gruppe darf alles, Others nur betreten         |
| Sticky Bit (`+t`) | Nur Datei-Owner darf eigene Dateien löschen    |
| SGID Bit (`g+s`)  | Neue Dateien erben automatisch Gruppe `www-data`|
