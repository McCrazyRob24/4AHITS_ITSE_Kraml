# Übung 1: Binärdaten und Hexdump-Analyse
 
## Ziel der Übung
 
Da kompilierte Programme Binärdaten enthalten und nicht nur gültige
ASCII-Zeichen, ist eine Anzeige mit `cat` nicht zielführend. In dieser
Übung wurde der Inhalt der Programmdatei `pwcheck` stattdessen als
Hexdump mit dem Tool `xxd` analysiert. Ein Hexdump zeigt dabei drei
Spalten: Offset, Hexcodes der Daten und die ASCII-Darstellung.
 
---
 
## Hexdump mit xxd
 
```sh
xxd pwcheck
```
 
Da die Ausgabe sehr umfangreich ist, wurde sie zusätzlich seitenweise
dargestellt:
 
```sh
xxd pwcheck | more
```
 
Da es sich bei `pwcheck` um eine Linux-Programmdatei handelt, kann
zusätzlich mit `objdump` mehr über den Inhalt (z. B. die Disassembly)
in Erfahrung gebracht werden.

# Übung 2: GNU Debugger – Passwortabfrage umgehen
 
## Ziel der Übung
 
In dieser Übung wurde das Programm `pwcheck` mittels Reverse
Engineering analysiert, um die Passwortabfrage zu umgehen. Reverse
Engineering bedeutet, eine bestehende Software zu analysieren, um deren
Funktionsweise zu verstehen. Nachdem die binäre Struktur des Programms
verstanden wurde, wurde dieses zur Laufzeit im Speicher mit dem GNU
Debugger (`gdb`) verändert (gepatcht).
 
---
 
## Programm im Debugger starten
 
```sh
gdb ./pwcheck
(gdb) b main
(gdb) run
```
 
Die Disassembly an der relevanten Stelle zeigte den Aufruf der
Passwortprüfung sowie den anschließenden bedingten Sprung:
 
```sh
0x00000000000011c0:  call   12a0 <_Z14check_passwordPKcS0_>
0x00000000000011c5:  cmp    $0x0,%eax
0x00000000000011c8:  je     1263 <main+0xf3>
```
 
Der Befehl `je` (jump if equal) springt nach erfolgreicher
Passwortprüfung zur eigentlichen Programmlogik. Wird dieser Sprung mit
`nop`-Befehlen (No Operation) überschrieben, wird die Prüfung
wirkungslos und das Programm läuft unabhängig vom eingegebenen
Passwort immer weiter.
 
---
 
## Patchen des Sprungbefehls
 
Die 6 Bytes des `je`-Befehls wurden im Speicher einzeln mit `nop`
(Opcode `0x90`) überschrieben:
 
```sh
(gdb) set {unsigned char}0x5555555551c8 = 0x90
(gdb) set {unsigned char}0x5555555551c9 = 0x90
(gdb) set {unsigned char}0x5555555551ca = 0x90
(gdb) set {unsigned char}0x5555555551cb = 0x90
(gdb) set {unsigned char}0x5555555551cc = 0x90
(gdb) set {unsigned char}0x5555555551cd = 0x90
(gdb) c
```
 
Die Passwortabfrage wurde dadurch erfolgreich umgangen, da der
bedingte Sprung nicht mehr ausgeführt wird.
 
---
 
## Wiederholung mit disas
 
Bei einem erneuten Start des Programms wurden mit `disas main`
zunächst die aktuellen Adressen ermittelt, bevor erneut gepatcht wurde:
 
```sh
gdb pwcheck
disas main
(gdb) set {unsigned char}0x00005555555551a5 = 0x90
(gdb) set {unsigned char}0x00005555555551a6 = 0x90
(gdb) set {unsigned char}0x00005555555551a7 = 0x90
(gdb) set {unsigned char}0x00005555555551a8 = 0x90
(gdb) set {unsigned char}0x00005555555551a9 = 0x90
(gdb) set {unsigned char}0x00005555555551aa = 0x90
```
 
Die Adressen unterschieden sich vom ersten Durchlauf, da PIE-Binaries
(Position Independent Executables) durch ASLR (Address Space Layout
Randomization) bei jedem Start an eine andere Basisadresse geladen
werden. Vor jedem Patch-Vorgang müssen die aktuellen Adressen daher
neu mit `disas` ermittelt werden.

# Übung 3: pwn – Addition zu Subtraktion patchen
 
## Ziel der Übung
 
Nach der Passwortabfrage liest das Programm zwei Zahlen ein und
addiert diese. Ziel dieser Übung war es, mit `gdb` die Binärdaten so
zu manipulieren, dass statt einer Addition eine Subtraktion
ausgeführt wird.
 
---
 
## Ausgangszustand (Addition)
 
```sh
(gdb) c
Continuing.
Enter password: test
Correct
Password was: G7pL9xQ2mR4tZc
Ha=7
b=7
Result=14
```
 
Das Programm addiert die beiden eingegebenen Werte (7 + 7 = 14)
korrekt.
 
---
 
## Patchen zu Subtraktion
 
Der Opcode für `add` wurde durch den Opcode für `sub` (`0x29`) ersetzt:
 
```sh
set {unsigned char}0x0000555555555224 = 0x29
```
 
```sh
(gdb) c
Continuing.
Enter password: test
Correct
Password was: G7pL9xQ2mR4tZc
Ha=7
b=1
Result=6
```
 
Nach dem Patch wurde statt der Addition korrekt eine Subtraktion
ausgeführt (7 − 1 = 6). Die Manipulation eines einzelnen Opcode-Bytes
genügt, um die Operation des Programms grundlegend zu verändern.

# Übung 4: Binary File Patching I
 
## Ziel der Übung
 
Die zuvor im Speicher durchgeführten Änderungen wurden in dieser
Übung permanent im Binärfile des Programms vorgenommen. Dazu wurde
zunächst mit `objdump -d pwcheck` der Offset der zu manipulierenden
Daten innerhalb der Datei ermittelt.
 
---
 
## Patchen mit hexedit
 
Die Datei wurde mit `hexedit` geöffnet und der ermittelte Offset
direkt angesprungen:
 
```sh
hexedit pwcheck
```
 
Vorgehensweise:
1. `Ctrl+G` (goto) — Offset in Hex eingeben
2. Bytes mit `90` (NOP) überschreiben
3. `Ctrl+X` (save and exit)
Vorher/Nachher-Vergleich der Disassembly:
 
```sh
11a2:       83 f8 00                cmp    $0x0,%eax
11a5:       90                      nop
11a6:       84 92 00 00 00 48       test   %dl,0x48000000(%rdx)
```
 
Im Vergleich zum ursprünglichen `je`-Befehl an Offset `11a5` zeigt die
Ausgabe nun einen `nop`-Befehl. Im Gegensatz zum Patchen im Speicher
mit `gdb` bleibt diese Änderung permanent in der Datei erhalten und
wird bei jedem weiteren Start des Programms automatisch wirksam.
 
---
 
## Zusammenfassung
 
In diesen Übungen wurde ein Programm mittels Reverse Engineering
analysiert und an mehreren Stellen gepatcht. Mit `xxd` und `objdump`
wurde der Binärinhalt einer Datei sichtbar gemacht, mit `gdb` wurden
gezielt einzelne Bytes im Arbeitsspeicher überschrieben, um
Sprungbefehle zu deaktivieren und arithmetische Operationen zu
verändern. Da Speicheradressen durch ASLR bei jedem Programmstart
variieren, mussten Patches im Speicher bei jedem Durchlauf neu
vorgenommen werden. Mit `hexedit` wurden dieselben Änderungen
schließlich permanent in die Binärdatei geschrieben, sodass sie bei
jedem Start automatisch aktiv sind. Diese Übung zeigt, wie wenig
Aufwand notwendig ist, um Sicherheitsmechanismen eines Programms durch
direkte Manipulation des Maschinencodes auszuhebeln.
